ACCOUNT / PROFILE PAGE MIGRATION AUDIT
======================================

Date: 2026-07-03
Scope: Current repository state on branch master
Status: Audit only. No account, navigation, privacy, subscription, or permission
logic was changed.

This is an engineering and store-readiness audit, not legal advice. Store
requirements and local law should be reviewed again immediately before release.


AUDIT SUMMARY
-------------

The current "You" experience is not an account system. It is a bottom sheet
containing hard-coded profile, permission, reporting, and subscription labels.

Repository evidence shows:

1. There is no implemented user authentication or account creation.
2. There is no Supabase Auth usage and no user/profile table in the generated
   database types.
3. There is no RevenueCat, StoreKit, Google Play Billing, Expo IAP, or Stripe
   client/server integration.
4. Favorites and garage vehicles are held only in React state. They are not
   persisted to disk or Supabase and disappear when the JavaScript app state is
   recreated.
5. Location permission is real, but YouBottomSheet always displays "Allowed"
   without checking the current permission.
6. "Anonymous reporting enabled" is hard-coded. The repository contains no
   reporting write path or device-attribution implementation.
7. "Munich pilot Active" is hard-coded. There is no subscription entitlement.
8. The Privacy row is not interactive and no in-app privacy policy, consent
   controls, retention policy, deletion policy, terms, or support link exists.
9. The billing screen is a nonfunctional mock. It describes Stripe checkout for
   digital premium functionality, but the CTA has no handler.
10. The existing `(tabs)/account.tsx` route is not a page. It opens the `you`
    overlay and redirects to `/map`.

Account deletion is not currently required by Apple/Google account-creation
rules because this codebase does not create accounts. A truthful "Delete local
data" or "Reset app data" flow is still appropriate. If authentication or an
automatically generated guest account is added later, in-app account deletion
becomes mandatory. Google Play will also require a usable outside-app deletion
web resource.

The clean migration is to keep `/account` inside the existing `(tabs)` Stack,
render a dedicated AccountScreen there, and navigate to it exactly as Garage is
navigated to. This preserves the persistent bottom navigation and access to the
FavoriteParkingProvider and VehicleProvider.

How You opens:

     key: profile
     label: YOU
     callback: onProfilePress

2. `(tabs)/_layout.tsx` passes:

     onProfilePress={() => showOverlay('you')}

3. showOverlay toggles the MapOverlayContext state. If the current route is not
   `/map`, it replaces the current nested route with `/map`.

4. `parking-map.tsx` renders YouBottomSheet when:

     activeOverlay === 'you'

5. Opening any non-none overlay clears selected parking/search state.

How You closes:

  - Tapping the close button calls `sheetRef.current?.close()`.
  - Swiping down closes the bottom sheet.
  - Tapping the YOU navigation item again toggles the overlay to `none`.
  - Android hardware back calls closeOverlay when an overlay is active.
  - The sheet's onClose callback calls closeOverlay.

How billing opens:

1. "Manage subscription" sets a mutable `shouldOpenBillingRef`.
2. The You sheet closes first.
3. Its onClose callback closes the overlay.
4. If the ref was set, it pushes `/billing`.

Current `/account` deep link:

1. AccountDeepLink mounts.
2. Its effect calls openOverlay('you').
3. It redirects to `/map`.
4. The map then renders YouBottomSheet.

Garage comparison:

  - `/garage` renders GarageScreen directly.
  - Bottom navigation remains visible because it is outside the nested Stack.
  - Garage provides a custom Back button.
  - The Back button uses router.back() when history exists and otherwise
    router.replace('/map').
  - Garage reads VehicleContext directly because the providers wrap the nested
    Stack.

CURRENT USER DATA FLOW
----------------------

1. Favorites

Owner:
  FavoriteParkingProvider

Stored fields:
  Full ParkingClusterResponse objects for selected favorite parking items.

Storage location:
  React useState only.

Persistence:
  None. No disk or backend persistence.

Deletion:
  Individual favorites can be removed in FavoriteParkingBottomSheet or toggled
  from parking detail. No bulk clear/reset API exists.

Lifetime:
  Survives navigation while the provider remains mounted. It is lost when the
  provider/app JavaScript state is recreated.


2. Garage / cars

Owner:
  VehicleProvider

Stored fields:
  - id generated from timestamp plus an in-memory sequence
  - nickname
  - normalized license plate
  - createdAt timestamp
  - optional updatedAt type field, currently not set by add
  - activeVehicleId

Storage location:
  React useReducer state only.

Persistence:
  None. No disk or backend persistence.

Deletion:
  Individual vehicles can be deleted from GarageScreen. Deleting the active
  vehicle selects the first remaining vehicle. No bulk clear/reset API exists.

Privacy classification:
  A license plate can be personal data. If vehicle persistence or backend sync
  is added, it must be covered by the privacy policy, retention/deletion flow,
  store disclosures, and account deletion where applicable.


3. Location permission and coordinates

Configuration:
  app.config.ts requests foreground coarse/fine location only. Background
  location and Android foreground-service location are disabled.

iOS purpose text:
  "Allow WhiteChocolate to use your location to show where you are and find
  nearby parking."

Android permissions:
  ACCESS_COARSE_LOCATION
  ACCESS_FINE_LOCATION

Runtime flow:
  - The app does not request location automatically on map mount.
  - The map starts at the Munich overview.
  - Tapping the current-location control checks existing foreground permission.
  - If permission can be requested, it requests permission.
  - It checks whether location services are enabled.
  - It requests one high-accuracy current fix with a 15-second timeout.
  - On failure it may use an OS-provided last-known position no older than two
    minutes and no less accurate than 500 meters.
  - Coordinates are stored in React state and projected as a map marker.

Network implication:
  After obtaining location, parking-map calls requestParkingForCamera with the
  device coordinates at zoom 17. That produces a geographic bounding box and
  sends the bounds to Supabase to retrieve nearby parking segments. The exact
  GPS object is not inserted into a user table, but a user-centered map area and
  the network request IP are processed by Supabase infrastructure. The privacy
  policy and Play Data safety analysis must account for the actual provider and
  server logging configuration.

Current account UI problem:
  YouBottomSheet always says "Allowed". It does not query whether the permission
  is granted, denied, restricted, or askable.

Missing controls:
  - No open-system-settings action.
  - No permission refresh after returning from Settings.
  - No in-app location explanation tied to actual current state.
  - No explicit explanation that nearby map bounds are sent to Supabase.

7. Device and session identifiers

Explicit stable user/device identifier:
  None found.

Identifiers that do exist:

    Generated from Date.now() plus Math.random(). It is sent to Google's Places

    Timestamp plus in-memory sequence. It is local runtime data, not a device or

    Identifies the application/public API role, not an individual user.

ACCOUNT DELETION DECISION
-------------------------

Current app:
  Account deletion is not currently required because no app account is created.

Current local-data responsibility:
  A clear "Delete local data" / "Reset app data" action is still recommended.
  It should clear:

  - Favorites
  - Vehicles
  - Active vehicle selection
  - Any future persisted consent preferences
  - Any future cached local profile/subscription metadata

It must state what it does not do:

  - It does not revoke the OS location permission.
  - It does not clear provider/server logs outside the app.
  - It does not cancel an App Store or Google Play subscription.
  - It is not "Delete account" because no account exists.

If account creation is added:

  Apple:
    The app must let every user initiate deletion of the full account and
    associated data in the app. Automatically created guest accounts are also
    covered. Deactivation alone is insufficient.

  Google Play:
    The app must provide a discoverable in-app account deletion path and a
    functional outside-app web resource where account and associated-data
    deletion can be requested.

  Both:
    Any retained data and legal/security retention reason must be explained.
    Subscription cancellation and account deletion must be treated as separate
    operations.

    or text in the app, including apps that collect no personal/sensitive data.
    The hosted policy must be public, active, non-geofenced, non-editable, and
    not a PDF. It must identify the app/developer, data practices, third parties,
    security, retention/deletion, and a privacy contact.

RISKS AND EDGE CASES
--------------------

Navigation:

  - Pressing YOU repeatedly can push duplicate account screens unless guarded.
  - Navigating from Garage should close stale map overlay state.
  - A cold deep link to `/account` may have no back history; use map fallback.
  - Android hardware back should be handled by the nested Stack once `you` is no
    longer an overlay.
  - Bottom content needs enough inset/padding to remain above BottomNavBar.

Provider lifetime:

  - AccountScreen must stay within `(tabs)` to access vehicle/favorite contexts.
  - Moving account to the root Stack would hide the bottom nav and may lose
    direct provider access.

Permission state:

  - Permission can change while the app is backgrounded in system Settings.
  - Distinguish denied-but-askable from permanently denied.
  - iOS limited/restricted and Android service-disabled states need clear copy.
  - Opening Settings does not guarantee permission changes.

Local reset:

  - Require confirmation.
  - Reset active vehicle when vehicles are cleared.
  - Do not label local reset as account deletion.
  - Do not claim it clears OS permission or provider logs.
  - If persistence is added later, ensure reset removes both memory and storage.

Subscription:

  - Account deletion does not automatically cancel store billing.
  - Hard-coded EUR pricing is wrong for other storefronts/taxes/currencies.
  - Purchase status must survive reinstall and work across the user's devices.
  - Restore and entitlement verification need offline/error states.
  - Mock purchase controls should not ship in a review build.

Privacy:

  - Provider logs and retention cannot be inferred solely from app source.
  - Google Places search text can be personal/sensitive depending on queries.
  - User-centered Supabase bounds may constitute location processing.
  - License plates become a disclosure/deletion concern once persisted.
  - Store console disclosures must match every release and third-party SDK.

Product identity:

  - Legal documents and deletion pages must use the same app/developer identity
    as the store listing.

Key architecture decisions:

  1. Keep the new account route inside `(tabs)`.
  2. Keep login optional while the product is anonymous.
  3. Show only real state.
  4. Separate local-data reset, account deletion, and subscription cancellation.
  5. Treat real store billing as a separate implementation project.
  6. Remove the old You sheet only after the full route is verified.
